ⓘAI 辅助写作声明：本文由博主构思逻辑，AI 辅助润色，双方共同校对。

这篇算是对 2025 复盘里“编辑器/渲染/多维表格”那条主线的补完：你会发现很多看似不相关的东西——图文编辑器、多维报表、各种 Workflow 编排器、IDE——最后都容易收敛到同一种架构：Core + Plugins。

它不是潮流，而是一种“工程重力”：当交互足够重、扩展足够多、协作足够复杂时，系统会逼你把“稳定的真相来源”和“变化的业务诉求”拆开。

换个更生活化的类比：Core 像插排的底座，Plugins 像各种各样的插头。底座的价值不在于它“功能强”，而在于它提供了一套统一的孔距、接地、过载保护、开关与保险丝——让你可以在不改墙内线路的前提下，持续往上加东西，同时不至于把整栋楼烧掉。

1. 先把名词讲清楚 ​

我这里说的 Core + Plugins，接近“微内核（Microkernel）/ 插件式架构”的直觉：

  • Core（内核）：只做最稳定、最通用、最难替换的那部分能力；它管理数据模型、生命周期、调度、约束与预算，并且掌握最终裁决权
  • Plugins（插件）：承接“变化”与“横向扩展”；它可以加能力、加规则、加交互、加视图，但必须在 Core 给定的约束里运行

更关键的是两条“看不见但最贵”的东西：

  • Pipeline（管线）：把一件事拆成几个稳定阶段，规定每个阶段允许做什么、不允许做什么
  • Schema（协议/模型）：定义系统“真相”的形状，以及允许的变换范围

如果只学到“Core + Plugins”，但没有 Pipeline 和 Schema，你得到的往往不是“可扩展”，而是“可失控”。

2. 为什么图文 / 表格 / Workflow 都会长成这样 ​

我用一个非常粗暴但好用的判断式：

当一个系统同时满足下面三点，它就很难不走向插件化：

  1. 交互特别重：大量输入事件、复杂状态机、强实时反馈，性能与一致性都有硬指标
  2. 横向扩展特别多：需求不是“加一个页面”，而是“加一类能力”，并且会持续长出新分支
  3. 多团队共建：不同团队在同一平台上做不同“能力包”，需要并行开发、灰度上线、快速回滚

这时你会遇到一个本质矛盾：
“变化”必须快，但“真相”必须稳。

把变化塞进核心，会导致核心变成杂物间；把真相放到插件里，会导致系统没有统一口径。Core + Plugins 的底层逻辑，就是用架构把这两者隔开：Core 负责统一口径和强约束，Plugins 负责把变化挂上去。

这也是为什么它常见于编辑器与 IDE：比如 ProseMirror 把绝大多数扩展做成插件（plugins），VS Code 把能力扩展做成 extension 并通过 extension host 隔离运行环境（见参考链接）。再往外看，Kubernetes 也把一堆核心能力“接口化”：网络（CNI）、存储（CSI）、运行时（CRI）——本质上也是把“机制”留在核心，把“策略/实现”交给插件生态。

3. Case：你其实已经每天在用插件架构了 ​

下面这些例子，背后都指向同一个事实：当系统必须长期增长，扩展边界就必须被制度化。

系统Core 负责什么Plugins 负责什么常见坑
ProseMirror文档状态、事务、Selection、插件调度输入规则、快捷键、装饰、协同接入非确定性、副作用、插件互相打架
VS Code工作台、文件树、编辑器框架、协议通道语言服务、主题、调试器、工作流扩展性能、权限边界、API 兼容
Eclipse/OSGi模块系统、生命周期、服务注册IDE 能力包、工具链、业务功能版本地狱、依赖冲突、启动性能
Kubernetes控制面、调度、对象模型CNI/CSI/CRI、Operator升级兼容、观测链路、接口语义漂移

这些例子里，Core 的共同点不是“代码行数少”，而是“它掌握不变量”：对象模型与生命周期、调度与预算、以及最终裁决权。

4. 这种方式的优点：你用约束换到了自由 ​

4.1 内聚：核心更“纯”，扩展更“像扩展” ​

插件化最直观的收益是内聚：
核心不再需要知道“某个业务想怎么搞”，它只要提供一套可组合、可校验、可调度的机制。

sequenceDiagram
    participant Core
    participant Plugin
    
    Core->>Plugin: init()
    Note right of Plugin: 注册能力 / 声明依赖
    
    loop Runtime
        Core->>Plugin: hook(context)
        Plugin-->>Core: result / side-effect
    end
    
    Core->>Plugin: destroy()
    Note right of Plugin: 清理资源 / 解绑事件

换句话说，核心代码的“纯粹”不是美学追求，而是工程生存：核心越像稳定的基础设施，它越能承受长期迭代与团队更替。

4.2 统一约束：所有扩展都必须遵守同一套规矩 ​

在重交互系统里，最贵的不是“实现功能”，而是“保证系统不坏”。

Core 能做的事情是：

  • 给插件统一的生命周期与挂载点（什么时候能读、什么时候能写、什么时候只能声明）
  • 给性能预算与降级策略（例如虚拟滚动的边界、绘制合批的策略、输入响应的延迟上限）
  • 给一致性约束（例如事务模型、撤销栈、协同一致性、回放可复现）

插件不需要发明自己的世界观，只需要在这套约束里把问题解决掉。

4.3 并行开发与灰度：组织层面的扩展能力 ​

插件化天然适合做“能力包”的并行交付：

  • 插件可以独立迭代、独立灰度、独立下线（理想情况）
  • Core 可以通过开关/白名单控制插件组合，从而做到“按用户、按场景”组合能力

对一个平台型产品来说，这就是组织级别的吞吐量。

5. 代价与坑：插件不是万能药 ​

5.1 复杂度转移：从“写功能”变成“管生态” ​

当你开放了插件，你就不再只写一个产品，而是在维护一个生态系统：

  • 插件版本兼容、能力探测、降级策略
  • 插件之间的冲突与优先级
  • API 演进的节奏：你改一次核心 API，可能影响一片插件

很多团队会在“插件化的前两个月”感受到自由，在“插件化的第二年”感受到债务。

5.2 性能与确定性：最容易被插件悄悄掏空 ​

插件最危险的地方，是它能在你看不见的地方做事：

  • 读外部世界（时间/随机数/网络）导致非确定性
  • 持有全局可变状态导致互相污染
  • 在高频阶段做重活导致卡顿，但 blame 很难归因

重交互系统里，“确定性”通常不仅是协同需要，也是灰度与排障需要：你只有能把某次输入在同一组插件下重放出来，才谈得上定位问题、稳定回归。

这一点可以换成一个更工程化的说法：插件越多，你越需要把系统变成“可追溯的纯函数”，哪怕它看起来一点也不纯。否则你会在事故现场听到那句最恐怖的话：
“我这边没复现啊。”

5.3 安全与合规：插件越强，平台越危险 ​

一旦插件能触达网络、文件系统、剪贴板、埋点上报，它就很容易绕开平台的风控边界。
很多平台最终都会把“访问外部世界”单独做成受控通道：可审计、可授权、可关停。

5.4 API 演进：核心每一次改动都会被生态放大 ​

插件架构最痛的，不是写第一版，而是写第三年：

  • 你不敢随便改 API，因为一堆插件依赖它
  • 你也不敢不改 API，因为核心需要演进

因此你会被迫引入“平台化工程”的那套东西：版本策略、Deprecation 周期、能力探测、文档与回归测试、以及真正可用的迁移工具。这些东西平时看起来都很“烦”，但一旦没有它们，生态会用更烦的方式惩罚你。

6. 维持这种架构，你真正需要的是什么 ​

我倾向于把答案浓缩成两句话：

  • 用强约束的 Pipeline，把副作用放在正确的层级
  • 用强定义的 Schema，把真相变成可校验、可演进的形状

6.1 Pipeline：把“能做什么”写死 ​

一个成熟的插件系统，通常会把核心流程拆成“几个很难被辩驳的阶段”：

flowchart LR
    Input[输入事件] --> Normalizer[归一化]
    Normalizer --> Transaction[事务构建]
    Transaction --> Validator[校验 & 约束]
    Validator --> State[应用到状态]
    State --> View[派生视图]
    View --> SideEffect[副作用 (网络/IO)]

    subgraph Plugins
        P1[插件 A]
        P2[插件 B]
    end

    P1 -.->|Hook| Transaction
    P2 -.->|Hook| View

    style State fill:#bbf,stroke:#333,stroke-width:2px

如果让我把它画得更“可执行”，大概会长这样：

flowchart LR
  In[输入事件] --> N[归一化]
  N --> Tr[事务/动作]
  Tr --> V[校验 & Normalize]
  V --> C[Commit]
  C --> R[渲染/派生视图]
  R --> S[副作用通道]

  P1[Plugins] -.hook.-> Tr
  P1 -.hook.-> V
  P1 -.hook.-> R

一个典型的约束长这样：

  • 有的阶段允许读写模型（但必须走事务/变换接口）
  • 有的阶段只允许声明（例如装饰、视图、指标）
  • 有的阶段允许异步，但必须把结果以显式数据写回（而不是在未来某个时刻偷偷改状态）

阶段划分的价值在于：它把争论从“你能不能这么写”变成“你现在处在哪个阶段”。规则清楚了，生态才可能长期可控。

6.2 Schema：把“真相是什么”写死 ​

Schema 的意义也不是“格式”，而是“不变量”：

  • 哪些字段必须存在
  • 哪些变更是合法的
  • 哪些东西只能由核心生成
  • 变更如何 normalize 与兼容演进

编辑器里，Schema 往往对应文档结构；多维表格里，Schema 往往对应字段/行/公式的模型；Workflow 里，Schema 对应节点/边/资源引用与运行时语义。

当 Schema 足够强，你才敢把扩展交出去：因为最终会有一个统一的校验与归一化层，把“可怕的自由”压回到“可控的自由”。

7. 适用边界：什么时候不该插件化 ​

我会克制地说：大部分系统都不需要插件化。

如果下面任意一点成立，就要非常谨慎：

  • 产品还在找 PMF，核心能力都没稳定
  • 扩展点很少，需求主要是纵向迭代而不是横向扩展
  • 团队规模小，沟通成本远小于架构成本

插件化的本质是“提前支付复杂度”，用架构换未来的并行与可控。未来如果不来，那就是白付。

8. 什么时候“应该”插件化：一个很土的决策清单 ​

我更愿意把它当成一套“自测题”，而不是架构信仰：

  • 是否有 10+ 个稳定存在的扩展点，而且会继续增长？
  • 是否有多个团队并行在同一平台上交付能力包？
  • 是否存在明确的性能预算/一致性约束（输入延迟、渲染预算、协同一致性、回放）？
  • 是否愿意为生态治理持续投入（版本策略、CI、兼容层、迁移工具）？

如果前两条不成立，插件化大概率是“提前过度设计”。
如果前三条成立但第四条不成立，插件化大概率会变成“平台烂尾”。

9. 小结：用约束换自由，用治理换规模 ​

Core + Plugins 看起来像技术架构，实际上更像一种工程契约：

  • Core 承诺提供稳定机制与强约束
  • Plugins 承诺在约束内做变化，并把副作用显式化

当它成立时，你会得到一个可以持续长大的系统；当它不成立时，你会得到一个“任何人都能改、但谁也不敢动”的大泥球。
