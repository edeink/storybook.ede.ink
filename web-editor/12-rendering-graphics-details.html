刚转岗飞书那会儿，我想补“渲染/图形学”这块短板。有山当时建议我去看 Games 101。很戏剧：他在我转岗当周就离职了，但这句推荐留了下来。我断断续续花了一年多把基础补起来，也发动人脉：邀请阿闷做过一次 图形学管线 分享，还抱上了 林铖的大腿。写这篇算是缅怀：很多人离开得很快，但他留下的一句话，会在你后面很长一段时间里持续影响。

问题：为什么 Web 编辑器要聊图形学？ ​

只做 DOM 渲染，很多能力是浏览器“免费送的”：字体栅格化、合成、滚动、选择、可访问性（a11y），

Hello World
，但凡接触前端的同学都能写。但也因为很多内容并没有标准化，导致存在各种样式差异。而一旦要求强一致性，高性能等，就不得不实现自绘制。而绘制的接口，可归类为：

  • Canvas 2D（为了批量绘制、可控的 overlay、或跨平台一致性）
  • WebGL/WebGPU（为了大量图元、特效、复杂装饰、或更稳定的 60FPS）
  • Hybrid（视觉层 GPU，语义层 DOM）

以上内容都和图形学密不可分。你就必须把“渲染”从一堆 API 调用，上升为一条可解释、可度量、可调优的管线。

渲染不是“谁更快”，而是“接管多少系统能力” ​

后端优势代价适合场景
HTML/CSS选区/输入法天然更稳，生态成熟超大规模节点与复杂装饰易掉帧常规富文本、结构语义优先
Canvas 2D批量绘制、实现成本中等你要自己做更多命中/选区/复制文档类渲染、白板、可视化
WebGLGPU 并发，适合大量图元与特效文字与排版工程复杂，调试门槛高图形/装饰密集、特效为主
WebGPU更现代的 GPU API，更可控的资源与管线生态与兼容仍在演进重渲染、未来方向、性能敏感

四舍五入：越往 GPU 走，越是“用性能换工程复杂度”。真正决定你能不能走下去的，是你是否有清晰的渲染模型与数据结构，而不是 API 选型。

从 1D 到 2D（再到有限 3D）：编辑器其实在做坐标系的同构 ​

写编辑器时，你会反复在几种“空间”之间来回跳：

  • 1D：纯文本/线性序列的空间，位置通常是一个偏移量
  • 2D：排版后的屏幕空间，位置是 (x,y)(x, y)(x,y)
  • 3D（有限）：如果你把 layer / z-index 也纳入模型，那就是 (x,y,z)(x, y, z)(x,y,z) 的分层空间

它们不是互斥的世界，更像是同一份内容在不同表述下的同构。很多“点不准/选区漂移/渲染抖动”的 bug，本质都是某条映射链路不闭合。

1D：纯文本为什么像一维？ ​

把一段内容抽象成线性序列，你得到的是一个最朴素的坐标：

p∈[0,N]p \in [0, N] p∈[0,N]

这里的 ppp 是逻辑位置（offset），NNN 是内容长度。纯文本编辑的很多操作，天然是在 1D 上做增删改查：insert/delete/splice。哪怕你底层用 Rope/Piece Table，接口层还是会暴露这种一维位置。

2D：布局为什么像二维？ ​

一旦你排版，你就需要把 1D 位置映射到 2D 几何：

L:p↦(x,y)L: p \mapsto (x, y) L:p↦(x,y)

反过来，交互（点击/拖拽）又需要把 2D 点位命中回 1D：

H:(x,y)↦pH: (x, y) \mapsto p H:(x,y)↦p

当 H(L(p))≈pH(L(p)) \approx pH(L(p))≈p 在工程上成立时，你的“光标、选区、命中”才会稳定。这里的“≈\approx≈”很现实：因为字形 cluster、半个 advance、连字、以及不同平台的渲染差异，你很多时候只能做到“足够接近且一致”。

3D（有限）：z-index 让你看起来进入三维 ​

严格来说，大多数编辑器并不需要真实的 3D 相机与深度测试；但你几乎一定会用到一个“有限第三维”：层级与覆盖顺序。

(x,y,z),z∈Z(x, y, z), \quad z \in \mathbb{Z} (x,y,z),z∈Z

这里的 zzz 更像是 layer id / z-index，而不是几何深度。渲染时你常用投影把它压回 2D：

π(x,y,z)=(x,y)\pi(x, y, z) = (x, y) π(x,y,z)=(x,y)

事件与命中时，你又会用 zzz 做“先后裁决”：同一个 (x,y)(x, y)(x,y) 命中多个对象时，选择 zzz 最大（或按绘制顺序最近）的那个。

四舍五入：从 1D 到 2D，再到带层级的“有限 3D”，都是同一件事——在不同空间里维护一致、可逆（或近似可逆）的映射。

坐标系与矩阵：把“玄学对齐”变回数学 ​

渲染与交互最常见的事故是坐标系混用。哪怕你只做 2D，也建议把变换统一成矩阵链路。

2D：仿射变换用 3×3（齐次坐标） ​

用齐次坐标可以把平移也写进矩阵：

[actxbdty001]⋅[xy1]=[x′y′1]\begin{bmatrix} a & c & t_x \\ b & d & t_y \\ 0 & 0 & 1 \end{bmatrix} \cdot \begin{bmatrix} x \\ y \\ 1 \end{bmatrix} = \begin{bmatrix} x' \\ y' \\ 1 \end{bmatrix} ​ab0​cd0​tx​ty​1​​⋅​xy1​​=​x′y′1​​

其中：

  • (a, b, c, d) 表示线性变换（缩放/旋转/错切）
  • (tx, ty) 表示平移

组合变换的核心是“乘法顺序”：你希望先缩放再平移，和先平移再缩放，结果不同。

只要你把每一层的矩阵定义清楚，很多问题都会变得可定位：

  • 画面缩放后，命中偏移：Inverse 用错或漏乘
  • DPR 下“看起来准、点起来不准”：渲染空间用了设备像素，事件空间用了 CSS 像素
  • 旋转/分页后选区错位：矩阵链路里缺了某一层的 transform

这类“链路要明确”的感觉，在很多渲染框架里都是一等公民。比如 Pixi.js 会在场景树里同时维护局部矩阵（local matrix）与世界矩阵（world matrix）：局部矩阵描述节点自身的平移/旋转/缩放，世界矩阵把父节点的变换叠上去，常见形式是：

Mworld=Mparent⋅MlocalM_{world} = M_{parent} \cdot M_{local} Mworld​=Mparent​⋅Mlocal​

当你需要把点从全局映射回局部（例如命中测试、拖拽、手柄交互），本质就是在用逆矩阵做坐标系切换。Flutter 也类似：渲染树/Layer 栈会把 Transform 这类变换用矩阵串起来，确保 local ↔ global 的坐标转换有据可依，而不是在各处散落一堆“手算偏移量”。

CPU vs GPU：差别不是“谁更快”，而是“怎么分工” ​

如果你把渲染理解为“把一堆顶点/像素算出来”，那你很容易误以为 CPU 和 GPU 是“同一类计算，只是谁快谁慢”。更接近事实的叙事是：它们分别沿着两条历史路径进化出来，目标函数不同。

  • CPU 的历史包袱与目标：更通用、更低延迟，擅长分支、复杂控制流、以及少量数据上的精细计算。为了把单线程逻辑跑快，CPU 发展出更深的流水线、更大的缓存层级、更强的乱序执行与分支预测。
  • GPU 的历史包袱与目标：更高吞吐，擅长把同一种简单计算在海量数据上“铺开”。图形管线天然带来大量向量/矩阵运算与像素级并行，GPU 因此走向更多的并行执行单元与更宽的 SIMD/SIMT 模型。

在编辑器这种“排版 + 渲染 + 交互”混合负载里，通常会得到一个很朴素的结论：

  • GPU 很适合做重复、规则、可批处理的计算（大量矩形、纹理采样、混合、简单矩阵变换）
  • CPU 很适合做不规则、强分支、强依赖、需要复杂数据结构支撑的计算（排版 shaping、命中索引、事务与映射、调度与增量更新）

一个更贴近工程的分工方式：

  • CPU 更擅长：排版（layout/shaping）、文本测量、构建 display list、合并批次、增量更新与调度
  • GPU 更擅长：大量重复图元的栅格化与混合、纹理采样、遮罩、特效、合成

GPU 管线的最小模型 ​

即使你不写复杂 shader，也建议掌握这个最小模型：

  1. 顶点（vertex）：几何形状的点集（矩形通常就是两个三角形）
  2. 顶点着色器（vertex shader）：把局部坐标变换到裁剪空间
  3. 光栅化（rasterization）：把三角形变成片元（fragment）
  4. 片元着色器（fragment shader）：决定每个像素的颜色（可能采样纹理）
  5. 混合与合成（blend/composite）：把结果叠到目标缓冲

对编辑器来说，大部分内容都可以归约为“矩形 + 纹理”的组合：

  • 文字：glyph atlas（纹理）+ quad（矩形）
  • 光标与选区：纯色矩形/圆角矩形
  • 图片：纹理 + quad
  • 下划线/批注高亮：路径或细矩形批次

Shader：你不一定要写，但你一定会被它影响 ​

WebGL（GLSL）与 WebGPU（WGSL）的工程差异 ​

WebGL 可以理解为 “OpenGL ES 的 Web 绑定”：WebGL 1.0 对齐的是 OpenGL ES 2.0 的能力模型，WebGL 2.0 对齐的是 OpenGL ES 3.0 的能力模型。它更偏“状态机”：你在一个上下文里不断 set state，然后 draw。

WebGPU 更像是 “Vulkan / Direct3D 12 / Metal 的 Web 版抽象”：它更偏“显式资源与管线”，你把渲染拆成更可控的对象（pipeline、bind group、command buffer），并且把大量隐式状态从“驱动内部”搬到“应用代码里”。

你用哪种 API 都可以，但工程上常见差异是：

  • WebGL：更“即时”，全局状态多，驱动要帮你做更多隐式工作（状态合并、验证、同步），容易写出“能跑但难治理”的渲染器
  • WebGPU：更“显式”，资源与管线更可控，能把很多成本前置到管线创建与命令录制阶段，代码更像真正的渲染引擎

一个很容易被误解但很关键的点是：WebGPU 的很多优化收益，首先体现在 CPU 侧，而不是 GPU 侧。

  • 更少的隐式状态与隐式同步：减少驱动猜测与校验开销
  • 更明确的资源生命周期：减少频繁的状态切换与上下文开销
  • 更接近“录制命令 → 提交”的模型：更利于把 CPU 工作拆分、并行化、批处理化

如果你做的是“复杂 UI 长期维护”，WebGPU 的显式模型往往更接近你想要的治理能力，但要面对兼容与生态的现实。

一个最常见的坑：预乘 alpha（premultiplied alpha） ​

做文本与 UI 混合时，经常会遇到边缘发黑/发白、颜色不对。最常见原因是纹理或管线在用预乘 alpha，但你的混合公式按非预乘去算（或反过来）。

如果你只记一个结论：

  • 统一全链路的 alpha 约定（纹理生成、上传、混合公式、输出缓冲）

不统一就会出现“偶现的边缘脏像素”，而这种问题特别难用日志解释。

文本渲染：最难的往往不是画，而是“像” ​

编辑器的文本渲染通常是“CPU 做 shaping，GPU 做绘制”的混合：

  • shaping：把 Unicode 字符序列变成 glyph 序列（连字、组合字符、方向性）
  • layout：决定每个 glyph 的位置与 advance
  • rasterize：把 glyph 变成位图（或 SDF/MSDF）
  • draw：把位图批量画出来

Glyph atlas：把“无数次小画”变成“少数次大画” ​

典型做法是维护一张或多张纹理 atlas：

  • 每个 glyph 占 atlas 的一个矩形区域
  • 渲染时只需要画很多个 quad，每个 quad 采样 atlas 的不同 UV

它的好处是能把 draw call 降到“按材质/纹理分批”，而不是“每个 glyph 一次”。

SDF/MSDF：让文字缩放更稳，但不是银弹 ​

SDF 思路：存储“到轮廓的距离”，shader 里做阈值与平滑，从而在不同缩放下保持边缘相对稳定。

再往上抽象一层：SDF 不只是“文字技巧”，它代表了一类很通用的几何/形状表示方式：用一个标量场去隐式描述边界。文字只是其中一个非常工程化、非常好落地的应用（轮廓清晰、边界定义明确、适合做 atlas）。

在更广的图形学世界里，常见的形状表示可以粗略分三类：

  • 显式表示：多边形网格（mesh）、点云（point cloud）等，数据就是几何本身
  • 隐式表示：SDF/occupancy 这类“场”，边界是某个等值面
  • 辐射场表示：NeRF 这类“视角相关的体渲染场”，关注的是光照/颜色沿射线的积分结果

在 2D UI/文本渲染里，SDF/MSDF 更常见；点云与 NeRF 这类表示更多出现在 3D 重建与体渲染问题里。把它们放在同一个坐标系里理解的价值是：当你开始做更复杂的渲染（例如模糊、描边、形变、动画），你会自然地遇到“形状表示”的选择题。

工程上常见取舍：

  • 小字号、需要 hinting 的场景：传统位图/系统字体栅格化更锐
  • 大字号、频繁缩放的场景：SDF/MSDF 更稳定

很多编辑器最终会走向“多策略并存”，而不是一把梭。

Tiling（瓦片化）：长文档与复杂装饰的救命稻草 ​

当内容很长、装饰很多、滚动频繁时，最危险的不是“单帧慢”，而是“每帧都在做重复工作”。瓦片化的核心直觉是：

  • 把大画面切成小块（tile）
  • 只重绘发生变化的 tile
  • 滚动时复用绝大多数缓存

典型的 tile cache 设计 ​

你可以把渲染缓存拆成两层：

  • display list：结构化的绘制指令（可重放、可裁剪）
  • tile cache：已经栅格化好的位图/纹理块（可复用）

然后用脏区（dirty rect）驱动增量更新：

  • 文档变更影响哪些块（由 layout/装饰器计算影响范围）
  • 哪些 tile 需要失效与重建
  • 哪些 tile 可以直接平移复用（滚动时最常见）

Tiling 与虚拟化的区别 ​

  • 虚拟化：减少参与渲染的“内容元素”
  • tiling：减少重复执行的“绘制工作”

两者常常一起用，但解决的问题不同：虚拟化偏内容层，tiling 偏像素层。

性能清单：最容易让帧率暴跌的几件事 ​

  • 频繁纹理上传：把动态内容拆分成局部纹理或走局部重绘
  • draw call 过多：批处理、合并同材质、减少状态切换
  • 缓存失效过大：让 layout/装饰器给出更精确的影响范围
  • 过度精细的遮罩/裁剪：优先 scissor/矩形裁剪，再考虑路径级 mask
  • 颜色空间与混合不一致：统一 gamma 与 alpha 约定，避免边缘脏像素
