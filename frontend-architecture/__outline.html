0. 范围 ​

  • 目标：把“架构能力”从概念落到可执行的决策与评审体系
  • 关注：可维护性、可扩展性、稳定性、性能、协作效率、成本与风险
  • 口径：不谈“架构很牛”，只谈“这个决策为什么值、哪里会翻车、怎么兜底”

1. 架构从哪来：先定义问题 ​

1.1 需求口径：先把问题写清楚 ​

  • 目标：业务目标与用户价值（不是“要一个页面”）
  • 约束：性能预算、兼容性、合规、交付周期
  • 风险：最可能翻车的三件事与预案
  • 指标：怎么衡量成功与失败
  • 回滚：回滚粒度、回滚策略与演练

1.2 分层与边界：复杂度要有收敛点 ​

  • 领域模型：把业务事实表达成稳定的对象与约束
  • 数据流：命令、事件与状态的关系
  • 渲染层：组件只是表现，别把业务塞进组件里
  • 基础设施：路由、请求、缓存、埋点、日志、权限

2. 状态管理与数据流 ​

2.1 状态分类：先区分“事实来源” ​

  • UI 状态：短生命周期、可丢弃、可重算
  • 领域状态：需要一致性与约束，适合集中管理
  • 服务端状态：缓存、重试、并发与一致性策略

2.2 数据流设计：能回放，才可治理 ​

  • 命令：意图明确、可校验、可拒绝
  • 事件：用于通知与观测，不要承载核心逻辑
  • 回放：把线上问题变成“可复现”而不是“凭感觉”

3. 组件体系与设计系统 ​

3.1 复用策略：把“复用”从口号变成成本模型 ​

  • 工具库：稳定、可测试、低耦合，优先沉到这里
  • 组件：承载交互与视觉，避免承载业务规则
  • hooks：抽离状态与副作用，但要警惕“隐式耦合”
  • 模板化：对齐结构与约束，降低沟通成本

3.2 设计系统：一致性来自约束 ​

  • 视觉 Token：颜色、字号、间距、暗黑模式
  • 交互规范：键盘操作、焦点管理、表单验证口径
  • a11y/i18n：从设计与组件契约阶段介入，而不是上线前补洞

4. 工程质量与稳定性 ​

4.1 测试：覆盖面与成本的平衡 ​

  • 单测：工具库与核心逻辑优先
  • 集成：模块边界与关键流程
  • E2E：关键链路兜底，避免把测试写成另一个产品

4.2 发布与灰度：把线上当成一个受控实验场 ​

  • 指标：上线前定义、上线后对齐
  • 报警：触发条件与分级，避免报警疲劳
  • 回滚：一键回滚、兼容旧数据与降级策略

5. 性能治理（工程化视角） ​

5.1 性能口径与预算：让“快”可衡量 ​

  • 指标：加载、交互、滚动、内存、长任务
  • 预算：每类页面/功能有不同预算，超预算必须有原因

5.2 防回归：性能是一个系统，不是一次优化 ​

  • 基准：代表性场景与固定设备口径
  • 看板：趋势 > 单点；对比 > 绝对值
  • 机制：变更影响评估、回归门禁与复盘闭环

6. 拟发布文章方向 ​

  • 架构评审清单：目标/约束/风险/指标/回滚的模板化
  • 状态与数据流：可回放系统的设计方法
  • 工程稳定性：从报警到复盘的完整闭环
  • 性能治理：预算化、门禁化与规模化落地
